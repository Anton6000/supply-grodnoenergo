Разработка веб-приложения для отдела снабжения филиалов РУП "Гродноэнерго" 

Дипломная работа содержит: 63 страницы, 23 иллюстрации, 20 использованных источников литературы.
Ключевые слова: Java, Spring Boot, React, PostgreSQL, REST API, веб-приложение, система снабжения, каталог номенклатуры, заявки на материалы, управление заказами, React Router, Spring Data JPA, Spring Security.
Цель исследования – разработка веб-приложения для цифровизации процессов снабжения подразделений РУП «Гродноэнерго» с реализацией централизованного каталога материалов и оборудования, модуля оформления и отслеживания заявок, а также административных и складских рабочих мест с использованием серверной части на Spring Boot, клиентской части на React и реляционной базы данных PostgreSQL.
Объект исследования: веб-приложение для автоматизации снабжения подразделений «Гродноэнерго», разработанное с использованием Java 21, Spring Boot 3.4.5, Spring Data JPA, Spring Security, Spring Validation, React 19, React Router 7, Create React App и PostgreSQL.
Предмет исследования: методы проектирования клиент-серверной архитектуры веб-приложения, реализация REST API для управления пользователями, каталогом и заказами, организация ролевой модели доступа (сотрудник подразделения, кладовщик, администратор), алгоритмы обработки жизненного цикла заявки от черновика до получения материалов, механизмы фильтрации и поиска по номенклатуре, валидация пользовательских данных при регистрации и аутентификации.
Методы исследования: на основе теоретического и сравнительного анализа существующих информационных систем учёта и снабжения на предприятиях энергетической отрасли была спроектирована архитектура одностраничного веб-приложения с модульной структурой и REST-сервисом на стороне сервера. Разработана серверная часть на Java с использованием Spring Boot и слоистой архитектуры (controller – service – repository – entity). Реализована реляционная модель данных, включающая сущности пользователей системы, товаров снабжения, заказов и позиций заказа. На клиентской стороне построен интерфейс на React с маршрутизацией по ролям пользователя и централизованным управлением состоянием приложения.
Данное приложение представляет собой комплексную платформу снабжения подразделений «Гродноэнерго» и включает:
•	каталог номенклатуры – просмотр материалов и оборудования с фильтрацией по категории, подкатегории, бренду, стране, ценовому диапазону, текстовым поиском и сортировкой;
•	модуль заявок – формирование корзины (черновика заказа), оформление заявки, отслеживание статусов («в ожидании», «принят», «в обработке», «собирается», «готов к получению», «получен»), подтверждение получения материалов;
•	личный кабинет сотрудника – управление профилем, просмотр активных заявок и истории заказов;
•	складской модуль – обработка входящих заявок, изменение статусов заказов, мониторинг рабочей области с автоматическим обновлением данных;
•	административный модуль – управление пользователями и номенклатурой, отображение ключевых показателей по заявкам;
•	модуль аутентификации – регистрация и вход сотрудников с проверкой учётных данных (электронная почта, табельный номер, пароль, роль).

ВВЕДЕНИЕ

В современных условиях цифровая трансформация затрагивает практически все сферы деятельности предприятий, включая материально-техническое снабжение. Для организаций энергетической отрасли, в том числе для подразделений РУП «Гродноэнерго», своевременное обеспечение материалами и оборудованием является важным условием бесперебойной работы. При этом традиционные способы организации снабжения всё чаще не отвечают требованиям оперативности, прозрачности и контроля.
На практике заявки на материалы нередко оформляются с использованием бумажных бланков, электронных таблиц, телефонных звонков и переписки по электронной почте. Такой подход увеличивает вероятность ошибок при вводе данных, затрудняет поиск нужной номенклатуры и не позволяет сотруднику оперативно узнать текущий статус заявки. Кладовщику также приходится вручную обрабатывать поступающие запросы, что замедляет работу склада и повышает административную нагрузку.
Существующие универсальные ERP-системы и корпоративные платформы закупок, как правило, рассчитаны на крупные предприятия и требуют значительных затрат на внедрение и сопровождение. Для отделов снабжения отдельных филиалов они часто оказываются избыточными по функционалу и недостаточно удобными в повседневной эксплуатации. В результате возникает потребность в специализированном веб-приложении, которое объединит каталог материалов, оформление заявок, контроль их статусов и администрирование справочной информации в единой системе.
Актуальность работы обусловлена необходимостью автоматизации процессов снабжения подразделений РУП «Гродноэнерго». Разработка собственного веб-приложения позволит сотрудникам формировать заявки через централизованный каталог, кладовщикам — обрабатывать заказы и изменять их статусы, администраторам — управлять пользователями и номенклатурой товаров. Это способствует сокращению времени обработки заявок, повышению прозрачности бизнес-процессов и снижению количества ошибок, связанных с ручным учётом.
Целью дипломной работы является разработка веб-приложения для автоматизации процессов снабжения подразделений РУП «Гродноэнерго». Приложение реализовано с использованием современных технологий: Java 21, Spring Boot 3.4.5, React 19, PostgreSQL и React Router 7.
В соответствии с поставленной целью были определены следующие задачи:
1.	Выполнить обзор и анализ существующих информационных систем учёта, закупок и снабжения.
2.	Сформировать функциональные требования к разрабатываемому веб-приложению.
3.	Разработать архитектуру системы, структуру проекта и выбрать технологический стек.
4.	Спроектировать и реализовать базу данных для хранения пользователей, номенклатуры, заказов и позиций заказа.
5.	Реализовать серверную часть на Spring Boot с REST API для управления каталогом, заявками и пользователями.
6.	Разработать клиентскую часть на React с модулями каталога, оформления заявок, личного кабинета, административной и складской панелей.
7.	Реализовать систему регистрации и аутентификации пользователей с ролевой моделью доступа.




















ГЛАВА 1
АНАЛИЗ ПРЕДМЕТНОЙ ОБЛАСТИ И ПОСТАНОВКА ЗАДАЧИ РАЗРАБОТКИ ВЕБ-ПРИЛОЖЕНИЯ ДЛЯ ОТДЕЛА СНАБЖЕНИЯ ФИЛИАЛОВ РУП «ГРОДНОЭНЕРГО»
1.1 Описание предметной области

Предметной областью настоящей работы является процесс материально-технического снабжения подразделений РУП «Гродноэнерго» — гродненского республиканского унитарного предприятия электроэнергетики, осуществляющего производство, передачу и распределение электрической и тепловой энергии в Принеманском регионе.
«Гродноэнерго» представляет собой единый технологический комплекс, включающий 14 филиалов — электростанции, сетевые и тепловые организации. Предприятие обеспечивает круглосуточное энергоснабжение региона, поэтому бесперебойная работа его подразделений напрямую зависит от своевременного обеспечения материалами, запасными частями и оборудованием.
Материально-техническое снабжение — это совокупность процессов, направленных на удовлетворение потребностей структурных подразделений в материальных ресурсах. Основные этапы включают формирование потребности, выбор позиций из номенклатуры, оформление заявки, её обработку, комплектацию на складе и выдачу материалов.
В процессе снабжения участвуют три категории пользователей. Сотрудники подразделений формируют заявки на материалы через каталог номенклатуры. Кладовщики принимают заявки в работу, контролируют комплектацию и изменяют статусы заказов. Администраторы ведут учёт пользователей и управляют справочником товаров.
Центральными элементами предметной области являются каталог номенклатуры и заявка на снабжение. Каталог содержит сведения о материалах и оборудовании: наименование, категорию, описание, бренд, единицу измерения и цену. Заявка формируется сотрудником путём выбора позиций и указания количества. До оформления она находится в статусе черновика, после отправки проходит этапы: «в ожидании» → «принят» → «в обработке» → «собирается» → «готов к получению» → «получен».
На практике процессы снабжения нередко организованы с использованием бумажных бланков, электронных таблиц и обмена информацией по телефону или электронной почте. Это приводит к задержкам в обработке заявок, ошибкам при переносе данных и отсутствию единого контроля статусов заказов.
Таким образом, предметная область охватывает автоматизацию процессов снабжения подразделений «Гродноэнерго»: ведение каталога, оформление и обработку заявок, управление их статусами и взаимодействие сотрудников, склада и администрации в рамках единой информационной системы.

1.3 Сравнительный анализ существующих систем

На основе проведённого обзора восьми белорусских интернет-магазинов электротоваров можно выделить общие закономерности в их организации и функциональности, а также существенные различия, определяющие область их применения.

Вторая группа — магазины с широким отраслевым ассортиментом: Intag.by, Mirelectriki.by, Avs.by и Elektroplus.by. Третья группа — универсальные торговые платформы, где электротовары составляют лишь часть каталога: Oma.by и 7745.by. Ни одна из рассмотренных систем не ориентирована на автоматизацию внутренних процессов снабжения предприятия.
По функциональности каталога большинство платформ обеспечивают базовые возможности: просмотр номенклатуры, фильтрацию по категориям, поиск по наименованию, отображение цены и описания товара. Наиболее развитые каталоги представлены у Avs.by, 7745.by и Oma.by — они предлагают широкую категоризацию и удобную навигацию. У Mirelectriki.by и Intag.by каталог также структурирован по разделам, однако ни одна из систем не предусматривает централизованное управление номенклатурой администратором предприятия с учётом внутренних потребностей организации.
По модели продаж все рассмотренные решения построены на классической схеме интернет-магазина: покупатель выбирает товар, оформляет заказ и оплачивает его. Оптовые закупки поддерживаются у Intag.by, Provodnik24.by, Elektroplus.by и Avs.by, однако это не заменяет корпоративный механизм формирования заявок сотрудниками подразделений. Отсутствует понятие «черновика заявки», внутреннего согласования и поэтапной обработки заказа в рамках одной организации.
По отслеживанию статусов заказа существующие системы предоставляют покупателю информацию лишь о стадии доставки или готовности к выдаче в рамках розничной покупки. Ни одна из восьми платформ не реализует полный жизненный цикл корпоративной заявки: «черновик → в ожидании → принят → в обработке → собирается → готов к получению → получен». Именно такая модель необходима для прозрачного взаимодействия между сотрудником подразделения, кладовщиком и администратором.
По ролевой модели доступа все обзорные системы ориентированы на единственную роль — покупателя. Отсутствует разграничение прав между сотрудником, кладовщиком и администратором, что является обязательным требованием для корпоративной системы снабжения. Административные функции (управление пользователями, ведение справочника номенклатуры, мониторинг заявок) в рассмотренных магазинах не предусмотрены.
По географическому охвату большинство платформ осуществляют доставку по всей Беларуси, включая г. Гродно (7745.by, Avs.by, Provodnik24.by, Elektroplus.by). Тем не менее региональное присутствие торговой точки не заменяет наличие единой цифровой платформы для всех подразделений предприятия.
По технической реализации все рассмотренные решения представляют собой готовые коммерческие интернет-магазины с закрытой архитектурой. Они не предоставляют возможности адаптации бизнес-логики под специфику конкретного предприятия, интеграции с внутренними учётными системами или развёртывания на собственной инфраструктуре организации.

Это обосновывает необходимость разработки специализированного веб-приложения для отдела снабжения подразделений РУП «Гродноэнерго», которое устранит выявленные недостатки и будет соответствовать специфике деятельности энергетического предприятия. 

1.4 Постановка задачи и требования к приложению

На основании анализа предметной области и сравнительного обозрения существующих интернет-магазинов электротоваров сформулирована задача разработки специализированного веб-приложения для автоматизации 
процессов снабжения подразделений РУП «Гродноэнерго».

Постановка задачи

Требуется создать клиент-серверное веб-приложение, которое обеспечит единую цифровую платформу для взаимодействия сотрудников подразделений, кладовщиков и администраторов в процессе заявок на материалы и оборудование. Система должна заменить разрозненные способы оформления заявок (бумажные бланки, электронные таблицы, обмен сообщениями) и предоставить централизованный каталог номенклатуры с возможностью формирования, обработки и контроля статусов заказов.
Приложение должно быть реализовано в виде одностраничного веб-приложения (SPA) на стороне клиента и REST API на стороне сервера с использованием реляционной базы данных PostgreSQL для хранения 
информации о пользователях, товарах и заявках.


Функциональные требования

1.	Управление пользователями. Система должна обеспечивать регистрацию и аутентификацию пользователей с разграничением ролей: сотрудник подразделения, кладовщик, администратор. Для входа используются электронная почта, табельный номер, пароль и роль пользователя.
2.	Каталог номенклатуры. Пользователь должен иметь возможность просматривать каталог материалов и оборудования с фильтрацией по категории, подкатегории, бренду, стране, ценовому диапазону, а также с текстовым поиском и сортировкой. Для каждой позиции отображаются наименование, описание, единица измерения, цена и изображение.
3.	Формирование заявок. Сотрудник подразделения должен иметь возможность добавлять товары в корзину (черновик заказа), изменять количество позиций, удалять их и оформлять заявку. До момента оформления заявка хранится в статусе «черновик».
4.	Обработка заявок. Кладовщик должен видеть все оформленные заявки и изменять их статусы по мере выполнения: «в ожидании», «принят», «в обработке», «собирается», «готов к получению». Сотрудник должен иметь возможность подтвердить получение материалов (статус «получен»).
5.	Личный кабинет сотрудника. Пользователь должен иметь доступ к профилю, просмотру активных заявок и истории заказов.
6.	Административные функции. Администратор должен управлять справочником пользователей (создание, редактирование, удаление) и номенклатурой товаров (добавление, изменение, удаление позиций каталога).
7.	Складской модуль. Кладовщик должен иметь отдельный интерфейс для просмотра и обработки входящих заявок с актуальной информацией о статусах заказов.

Нефункциональные требования

1.	Архитектура. Приложение реализуется по схеме «React SPA + Spring Boot REST API + PostgreSQL» с разделением на клиентскую и серверную части.
2.	Технологический стек. Серверная часть — Java 21, Spring Boot 3.4.5, Spring Data JPA, Spring Security, Spring Validation. Клиентская часть — React 19, React Router 7. База данных — PostgreSQL.
3.	Интерфейс. Пользовательский интерфейс должен быть понятным и адаптированным под рабочие задачи каждой роли. После входа пользователь перенаправляется на соответствующий раздел: каталог (сотрудник), складская панель (кладовщик), административная панель (администратор).
4.	Доступность. Приложение должно работать в современных браузерах и обеспечивать взаимодействие клиентской и серверной частей через REST API в формате JSON.
5.	Расширяемость. Архитектура системы должна допускать дальнейшее развитие: добавление новых категорий товаров, расширение статусной модели, интеграцию с учётными системами предприятия.
1.5	 Выводы по главе 1

В первой главе рассмотрена предметная область, проанализированы существующие решения и сформулированы требования к разрабатываемому веб-приложению для отдела снабжения подразделений РУП «Гродноэнерго».
Установлено, что материально-техническое снабжение подразделений энергетического предприятия включает формирование потребности в материалах, выбор позиций из номенклатуры, оформление и обработку заявок, комплектацию на складе и выдачу материалов. В процессе участвуют сотрудники подразделений, кладовщики и администраторы, каждый из которых выполняет определённые функции и нуждается в отдельном интерфейсе системы.
Проведённый обзор одиннадцати белорусских интернет-магазинов электротоваров показал, что на рынке представлены как узкоспециализированные, так и универсальные торговые платформы с развитым каталогом, доставкой по Беларуси и возможностью оптовых закупок. 
Сравнительный анализ выявил, что существующие системы ориентированы на розничную и оптовую продажу и не решают задачи внутреннего корпоративного снабжения. В них отсутствуют централизованное управление номенклатурой предприятия, механизм формирования заявок сотрудниками, ролевая модель доступа, контроль жизненного цикла заказа и взаимодействие между подразделением, складом и администрацией.
На основе проведённого анализа сформулирована задача разработки специализированного веб-приложения с каталогом материалов, модулем заявок, личным кабинетом, административной и складской панелями. Определены функциональные требования к системе (управление пользователями, каталог, заявки, обработка заказов, ролевой доступ) и нефункциональные требования (архитектура «React + Spring Boot + PostgreSQL», REST API, удобный интерфейс, расширяемость).
Таким образом, теоретический анализ подтверждает актуальность и необходимость разработки собственного веб-приложения для автоматизации процессов снабжения подразделений РУП «Гродноэнерго».

ПРОЕКТИРОВАНИЕ ВЕБ-ПРИЛОЖЕНИЯ ДЛЯ АВТОМАТИЗАЦИИ ПРОЦЕССОВ СНАБЖЕНИЯ 

2.1 Выбор технологического стека
                                                           
	Выбор технологий для разрабатываемого веб-приложения определялся требованиями к архитектуре клиент-серверной системы, необходимостью надёжного хранения данных, удобства сопровождения и соответствия современным подходам к разработке корпоративных информационных систем.

	Серверная часть

	В качестве языка программирования серверной части выбран Java 21. Java широко применяется при создании корпоративных приложений, обладает развитой экосистемой библиотек и фреймворков, обеспечивает стабильность, масштабируемость и переносимость кода. Использование актуальной версии языка позволяет применять современные возможности Java при разработке REST-сервисов.
	Основой серверной части является фреймворк Spring Boot 3.4.5. Spring Boot упрощает конфигурацию и развёртывание приложения, предоставляет готовые решения для создания REST API, работы с базой данных, обеспечения безопасности и валидации данных. Выбор Spring Boot обусловлен его распространённостью в корporate-разработке и удобством построения многослойной архитектуры (controller – service – repository – entity).
	Для взаимодействия с базой данных используется Spring Data JPA, который позволяет работать с реляционными данными через объектно-реляционное отображение (ORM) и сокращает объём шаблонного кода при реализации репозиториев. Доступ к данным организован через Hibernate с автоматическим созданием и обновлением структуры таблиц (ddl-auto=update).
	Обеспечение безопасности и контроля доступа реализуется с помощью Spring Security. В учебной версии проекта основная проверка пользователя выполняется на уровне API аутентификации, а Spring Security используется для настройки политики доступа и CORS.
	Валидация входных данных выполняется с применением Spring Validation, что позволяет проверять корректность данных при регистрации пользователей, создании товаров и оформлении заявок на уровне серверной части.

	База данных

	В качестве системы управления базами данных выбрана PostgreSQL. PostgreSQL — надёжная реляционная СУБД с открытым исходным кодом, поддерживающая транзакции, целостность данных и сложные запросы. Она подходит для хранения связанных сущностей системы: пользователей, товаров снабжения, заказов и позиций заказа. Использование реляционной модели данных обеспечивает согласованность информации и возможность 
построения связей между таблицами.

	Клиентская часть

	Пользовательский интерфейс реализован с использованием библиотеки React 19. React позволяет создавать одностраничные приложения (SPA) на основе компонентного подхода, обеспечивает эффективное обновление интерфейса и удобную организацию кода. Компонентная архитектура React хорошо сочетается с модульной структурой разрабатываемой системы (каталог, заявки, личный кабинет, административная и складская панели).
	Для навигации между страницами приложения применяется React Router 7, который обеспечивает маршрутизацию без перезагрузки страницы и реализует перенаправление пользователя в зависимости от его роли после 
входа в систему.

	Взаимодействие компонентов

	Клиентская и серверная части взаимодействуют через REST API в формате JSON по протоколу HTTP. Такой подход обеспечивает слабую связанность компонентов системы: frontend отвечает за отображение данных и работу пользователя, backend — за бизнес-логику и доступ к базе данных. В режиме разработки клиентское приложение работает на порту 3000, серверное — на порту 8080, обмен данными настраивается с учётом политики CORS.
Таким образом, сочетание Java 21, Spring Boot, PostgreSQL, React и React Router является обоснованным решением для реализации веб-приложения автоматизации процессов снабжения подразделений РУП «Гродноэнерго».

2.2 Архитектура приложения

Разрабатываемое веб-приложение для автоматизации процессов снабжения подразделений РУП «Гродноэнерго» построено по архитектуре клиент–сервер с разделением на три основных уровня: клиентская часть (presentation layer), серверная часть (business logic layer) и уровень данных (data layer).

Общая схема архитектуры

Клиентская часть представляет собой одностраничное приложение (SPA) на React, которое выполняется в браузере пользователя. Серверная часть реализована как REST-сервис на Spring Boot и обрабатывает HTTP-запросы от клиента. Уровень данных обеспечивается реляционной базой данных PostgreSQL, с которой серверная часть взаимодействует через Spring Data JPA.
Обмен данными между клиентом и сервером осуществляется по протоколу HTTP в формате JSON. Клиент отправляет запросы к REST API (/api/*), сервер выполняет бизнес-логику, обращается к базе данных и возвращает результат. Такой подход обеспечивает слабую связанность компонентов: клиентская и серверная части могут разрабатываться и 
развёртываться независимо друг от друга.

Архитектура серверной части

Серверная часть организована по многослойной архитектуре и 
разделена на функциональные модули:

•	client — регистрация, аутентификация и управление пользователями;
•	supply — каталог номенклатуры товаров снабжения;
•	order — формирование, обработка и контроль статусов заявок;
•	config — конфигурация безопасности, CORS и статических ресурсов.

Каждый модуль включает типовые слои:

1.	Controller — приём HTTP-запросов и формирование HTTP-ответов;
2.	Service — реализация бизнес-логики (оформление заявок, смена статусов, фильтрация каталога);
3.	Repository — доступ к данным через Spring Data JPA;
4.	Entity — сущности, отображаемые на таблицы базы данных;
5.	DTO — объекты передачи данных между слоями и при обмене с клиентом.

Архитектура клиентской части

Клиентская часть построена на компонентном подходе React и включает:

•	страницы (pages) — основные экраны приложения;
•	контекст (AppDataContext) — централизованное хранение состояния пользователя и данных приложения;
•	утилиты (utils) — вспомогательные функции для работы с API, валидации и форматирования данных;
•	маршрутизацию (React Router) — навигация между разделами без 
перезагрузки страницы.

Основные маршруты приложения:

Маршрут	Назначение
/	Главная страница
/catalog	Каталог номенклатуры
/catalog/product/:productId	Карточка товара
/sign-in-sign-up	Вход и регистрация
/profile/:id	Личный кабинет
/current-order/:id	Корзина и оформление заявки
/admin-dashboard	Административная панель
/warehouse-dashboard	Складская панель

После входа пользователь перенаправляется в раздел, соответствующий его роли: сотрудник — в каталог, кладовщик — на складскую панель, администратор — в административную панель.

Взаимодействие компонентов системы

Типовой сценарий работы системы:

1.	Пользователь выполняет действие в интерфейсе React (например, добавляет товар в корзину).
2.	Клиентская часть отправляет HTTP-запрос к REST API серверной части.
3.	Controller принимает запрос и передаёт его в Service.
4.	Service выполняет бизнес-логику и обращается к Repository.
5.	Repository выполняет операции с базой данных PostgreSQL.
6.	Результат возвращается по цепочке Service → Controller → клиент в формате JSON.
7.	React обновляет интерфейс на основе полученных данных.

Модульная структура системы

Функционально приложение состоит из следующих модулей:

•	модуль аутентификации — регистрация и вход пользователей;
•	модуль каталога — просмотр, поиск и фильтрация номенклатуры;
•	модуль заявок — корзина, оформление и отслеживание заказов;
•	модуль личного кабинета — профиль, активные заявки и история;
•	административный модуль — управление пользователями и товарами;
•	складской модуль — обработка заявок и изменение статусов.

Каждый модуль реализован как на клиентской (отдельные страницы и компоненты), так и на серверной стороне (отдельные контроллеры и сервисы), 
что обеспечивает модульность и расширяемость системы.

Преимущества выбранной архитектуры

Выбранная архитектура обеспечивает:

•	чёткое разделение ответственности между клиентом, сервером и базой данных;
•	возможность независимого развития frontend и backend;
•	масштабируемость за счёт модульной структуры;
•	удобство тестирования отдельных слоёв и модулей;
•	соответствие требованиям, сформулированным в первой главе.

Таким образом, архитектура приложения представляет собой классическую клиент–серверную модель с REST API и реляционной базой данных, адаптированную под задачи автоматизации процессов снабжения подразделений РУП «Гродноэнерго».

2.3 Проектирование базы данных.

Для хранения данных разрабатываемого веб-приложения выбрана реляционная СУБД PostgreSQL. Она обеспечивает целостность данных, поддержку транзакций и удобство работы со связанными сущностями.
Структура базы данных спроектирована на основе предметной области 
и включает четыре основные таблицы: system_users, supply_goods, orders и order_lines.Связь между таблицами реализована через внешние ключи, что позволяет однозначно определять, какой пользователь оформил заявку, какие 
товары в неё входят и в каком количестве.

Таблица system_users

Хранит сведения о пользователях системы. 

Поле	Назначение
user_id	Первичный ключ (формат: SU-1001, AD-1001, ST-1001)
full_name	ФИО пользователя
position	Должность
tab_number	Табельный номер (формат T-1111)
phone	Телефон
email	Электронная почта
password	Пароль
created_at	Дата регистрации
role	Роль: client, storekeeper, admin

Префикс идентификатора зависит от роли: SU- — сотрудник, ST- — кладовщик, AD- — администратор. Для полей ФИО, должности, табельного номера, телефона, email и пароля задано ограничение уникальности, что 
исключает дублирование учётных записей.

Таблица supply_goods

Содержит номенклатуру материалов и оборудования.

Поле	Назначение
supply_good_id	Первичный ключ (формат SG-1001)
good_name	Наименование товара
image_url	Путь к изображению
category	Категория
subcategory	Подкатегория
description	Описание
brand	Бренд
country	Страна производства
unit	Единица измерения (шт., м и т.д.)
price	Цена

Таблица является справочником каталога и используется при формировании заявок. Категория и подкатегория поддерживают фильтрацию и группировку товаров в интерфейсе.

Таблица orders

Хранит заявки (заказы) пользователей. 

Поле	Назначение
order_id	Первичный ключ (формат OR-1001)
user_id	Ссылка на пользователя из system_users
status	Статус заявки
created_at	Дата создания
updated_at	Дата последнего изменения

Статус заявки отражает этап обработки: draft (черновик), «в ожидании», «принят», «в обработке», «собирается», «готов к получению», «получен». Поле updated_at фиксирует момент последнего изменения статуса или состава заказа.

Таблица order_lines

Содержит позиции заявки — конкретные товары и их количество.

Поле	Назначение
order_line_id	Первичный ключ (формат OL-1001)
order_id	Ссылка на заявку из orders
supply_good_id	Ссылка на товар из supply_goods
quantity	Количество
unit_price	Цена на момент добавления в заявку

Поле unit_price фиксирует цену товара на момент оформления заявки. Это важно, поскольку цена в каталоге может измениться, а в уже оформленной 
заявке должна сохраняться первоначальная стоимость позиции.

Связи между таблицами

•	один пользователь (system_users) может иметь много заявок (orders) — связь один-ко-многим по полю user_id;
•	одна заявка (orders) может содержать много позиций (order_lines) — связь один-ко-многим по полю order_id;
•	один товар (supply_goods) может входить в много позиций разных заявок — связь один-ко-многим по полю supply_good_id.

Такая структура соответствует третьей нормальной форме: данные о пользователях, товарах и заявках хранятся отдельно, а позиции заказа 
связывают заявку с конкретными товарами без дублирования информации.

Особенности проектирования

При проектировании базы данных учтены следующие принципы:

1.	Идентификация записей — строковые идентификаторы с префиксами (SU-, SG-, OR-, OL-) упрощают чтение данных и соответствуют логике предметной области.
2.	Целостность данных — обязательные поля (nullable = false), уникальные ограничения для учётных данных пользователей, внешние ключи между связанными таблицами.
3.	Фиксация цены — хранение unit_price в order_lines обеспечивает корректный расчёт суммы заявки независимо от последующих изменений цены в каталоге.
4.	Временные метки — поля created_at и updated_at позволяют 
отслеживать историю создания и изменения заявок.

Таким образом, спроектированная база данных обеспечивает хранение пользователей, номенклатуры, заявок и позиций заказа, поддерживает связи между сущностями и соответствует функциональным требованиям системы снабжения подразделений РУП «Гродноэнерго».

2.4 Проектирование пользовательского интерфейса

Пользовательский интерфейс разрабатываемого веб-приложения спроектирован с учётом специфики корпоративной системы снабжения и потребностей трёх категорий пользователей: сотрудника подразделения, кладовщика и администратора. Основная задача интерфейса — обеспечить понятный и эффективный доступ к функциям системы без избыточной 
перегрузки элементами управления.

Принципы проектирования интерфейса

При проектировании пользовательского интерфейса были определены следующие принципы:

1.	Ролевая ориентация — каждая категория пользователей получает доступ только к тем разделам, которые соответствуют её задачам. После входа сотрудник перенаправляется в каталог, кладовщик — на складскую панель, администратор — в административную панель.
2.	Единый корпоративный стиль — интерфейс оформлен в фирменной стилистике «Гродноэнерго»: используются лаконичная цветовая палитра, логотип предприятия и единообразное оформление страниц.
3.	Минимум действий для выполнения задачи — оформление заявки сводится к последовательности «каталог → корзина → оформление», обработка заявки на складе — к просмотру списка и смене статуса.
4.	Наглядность статусов — каждый этап обработки заявки отображается текстовым статусом и визуальным индикатором прогресса, что позволяет пользователю быстро оценить текущее состояние заказа.
5.	Единое состояние приложения — данные пользователя, корзины и заявок хранятся в централизованном контексте (AppDataContext), что 
обеспечивает согласованность информации на всех страницах.

Структура интерфейса

Интерфейс приложения включает следующие основные экраны:

Главная страница (/) — информационный лендинг с описанием платформы снабжения «Гродноэнерго», сведениями о предприятии и сводкой каталога номенклатуры (категории, подкатегории, количество товаров). С главной страницы пользователь может перейти к входу в систему.
Страница входа и регистрации (/sign-in-sign-up) — объединённый экран аутентификации с переключением между формами входа и регистрации. На странице отображаются ключевые возможности системы: оформление заявок, складская обработка, администрирование. Форма входа запрашивает email, табельный номер, пароль и роль пользователя.
Каталог номенклатуры (/catalog) — основной рабочий экран сотрудника подразделения. Содержит список товаров с фильтрами по категории, подкатегории, бренду, стране, ценовому диапазону, полем поиска и сортировкой. Каждая карточка товара отображает изображение, наименование, цену и единицу измерения.
Карточка товара (/catalog/product/:productId) — подробная информация о позиции каталога: описание, характеристики, цена, кнопка добавления в корзину.
Страница заявки / корзины (/current-order/:id) — отображает текущий черновик заявки: список выбранных товаров, количество, сумму по каждой позиции и общую стоимость. Предусмотрены изменение количества, удаление позиций и оформление заявки.
Личный кабинет (/profile/:id) — профиль сотрудника с данными учётной записи, списком активных заявок и историей заказов. Для заявок в статусе «готов к получению» доступна функция подтверждения получения материалов.
Складская панель (/warehouse-dashboard) — рабочее место кладовщика. Отображает все оформленные заявки с информацией о заказчике, составе заказа и текущем статусе. Кладовщик может изменять статус заявки по мере её обработки. Данные обновляются автоматически через заданный интервал.
Административная панель (/admin-dashboard) — интерфейс администратора для управления пользователями и номенклатурой товаров. Включает таблицы со списком пользователей и товаров, формы создания и редактирования записей, а также сводные показатели по заявкам.


Навигация и маршрутизация

Навигация между экранами реализована с помощью React Router без перезагрузки страницы. Общая навигационная панель присутствует на большинстве страниц и содержит ссылки на каталог, корзину, личный кабинет и выход из системы. Состав доступных пунктов меню зависит от роли пользователя.

Элементы интерфейса по ролям

Для наглядного отображения этапов обработки заявки каждому статусу соответствуют текстовая метка и цветовой индикатор:

Такое представление позволяет и сотруднику, и кладовщику быстро определить текущий этап обработки заявки.

Формы регистрации и входа содержат поля с масками ввода (табельный номер, телефон) и проверкой данных на стороне клиента перед отправкой на сервер. При ошибках пользователю отображаются понятные сообщения (неверный пароль, занятый email, некорректный формат данных).

Иконки и оформление

Для улучшения восприятия интерфейса используется библиотека иконок Font Awesome. Иконки применяются в навигации, кнопках действий и информационных блоках, что делает интерфейс более наглядным без перегрузки текстом.
Таким образом, спроектированный пользовательский интерфейс обеспечивает удобную работу всех категорий пользователей системы снабжения, поддерживает ключевые бизнес-процессы предприятия и соответствует принципам проектирования корпоративных веб-приложений.

2.5 Проектирование бизнес-процессов и жизненного цикла заявки

При проектировании веб-приложения для снабжения подразделений РУП «Гродноэнерго» необходимо было формализовать бизнес-процесс оформления и обработки заявок на материалы. В системе участвуют три категории пользователей: сотрудник подразделения, кладовщик и администратор. Заявка проходит несколько последовательных этапов, каждый из которых имеет свой статус и определённого исполнителя.
Процесс начинается с того, что сотрудник входит в систему и переходит в каталог номенклатуры, где выбирает нужные материалы или оборудование. Выбранные позиции добавляются в корзину, которая в системе представлена как черновик заявки со статусом draft. На этом этапе заявка ещё не поступает на склад: сотрудник может добавлять товары, изменять количество и удалять позиции. Для каждого пользователя поддерживается один активный черновик, что упрощает формирование заявки.
После проверки состава сотрудник оформляет заявку. Её статус меняется с draft на «в ожидании», и она становится доступной для обработки кладовщику. С этого момента сотрудник может отслеживать ход выполнения заявки в личном кабинете, просматривать активные заказы и историю ранее оформленных заявок.
Дальнейшая обработка выполняется на складской панели. Кладовщик принимает заявку в работу и последовательно изменяет её статус: «принят» — заявка зарегистрирована; «в обработке» — выполняется обработка; «собирается» — идёт комплектация материалов; «готов к получению» — заявка собрана и готова к выдаче. Каждое изменение статуса фиксируется в системе, что обеспечивает прозрачность процесса для всех участников.
Завершающим этапом является получение материалов сотрудником подразделения. Когда заявка переведена в статус «готов к получению», сотрудник подтверждает факт получения в личном кабинете, после чего статус изменяется на «получен» и заявка считается завершённой. Таким образом, спроектированный жизненный цикл заявки охватывает весь процесс снабжения — от выбора товара в каталоге до выдачи материалов — и определяет логику работы разрабатываемой системы.

3.2 Создание и настройка базы данных PostgreSQL

После инициализации проекта и настройки среды разработки следующим этапом реализации стало создание и настройка базы данных PostgreSQL. База данных является центральным компонентом системы снабжения: в ней хранятся сведения о пользователях, номенклатуре товаров, заявках и позициях заказов. Корректная организация данных обеспечивает целостность информации, согласованность операций и надёжную работу REST API серверной части.

Установка и подготовка PostgreSQL

Для хранения данных приложения была установлена СУБД PostgreSQL. После установки был создан экземпляр сервера базы данных и база postgres, к которой подключается backend-приложение. 
Подключение настроено в файле application.properties:
spring.datasource.url=jdbc:postgresql://localhost:5432/postgres
spring.datasource.username=postgres
spring.datasource.password=root
spring.datasource.driver-class-name=org.postgresql.Driver
spring.jpa.hibernate.ddl-auto=update
spring.jpa.show-sql=true

Параметр spring.jpa.hibernate.ddl-auto=update указывает Hibernate автоматически создавать и обновлять структуру таблиц при запуске серверного приложения на основе JPA-сущностей. Это удобно на этапе разработки: при изменении entity-классов структура БД обновляется без ручного написания SQL-скриптов. Параметр spring.jpa.show-sql=true включает вывод SQL-запросов в консоль, что помогает при отладке.

3.3 Реализация серверной части на Spring Boot

Реализация серверной части веб-приложения выполнена на базе фреймворка Spring Boot 3.4.5 с использованием многослойной архитектуры. Серверная часть отвечает за обработку HTTP-запросов от клиентского приложения, выполнение бизнес-логики, взаимодействие с базой данных PostgreSQL и формирование ответов в формате JSON. Вся логика организована в пакете com.shop.ElectronicShop и разделена на функциональные модули: client, supply, order и config, что соответствует проектным решениям, описанным во второй главе.

Общая организация серверной части

Каждый функциональный модуль построен по единому принципу и
включает следующие слои. Слой controller принимает HTTP-запросы, извлекает параметры и тело запроса, вызывает соответствующий сервис и возвращает HTTP-ответ. Слой service содержит бизнес-логику: проверку данных, выполнение операций над сущностями, формирование результатов. Слой repository обеспечивает доступ к базе данных через интерфейсы Spring Data JPA. Слой entity описывает JPA-сущности, отображаемые на таблицы PostgreSQL. Слой dto (Data Transfer Object) определяет структуру данных, передаваемых между клиентом и сервером, и отделяет внешнее представление от внутренней модели.
Такое разделение позволяет изменять бизнес-логику без затрагивания слоя представления, упрощает тестирование отдельных компонентов и обеспечивает единообразие кода во всех модулях приложения. Для внедрения зависимостей используется механизм Spring IoC: сервисы получают репозитории через конструктор с аннотацией @RequiredArgsConstructor от Lombok, контроллеры — соответствующие сервисы. Транзакционность операций обеспечивается аннотациями @Transactional и @Transactional(readOnly = true) для операций чтения.

Модуль управления пользователями (client)

Модуль client реализует регистрацию, аутентификацию и административное управление пользователями системы. Основные компоненты модуля: ClientAuthController, SystemUserListController, ClientAuthService, ClientRegistrationService, ClientRepository и JPA-сущность Client, отображаемая на таблицу system_users.
Аутентификация пользователей реализована в классе ClientAuthService. Метод authenticate принимает запрос с email, табельным номером, паролем и ролью, находит пользователя в базе данных и проверяет совпадение всех учётных данных. Табельный номер нормализуется с помощью утилиты TabNumberNormalizer, что позволяет корректно сравнивать номера независимо от формата ввода. При успешной проверке возвращается ответ с ФИО, идентификатором и ролью пользователя; при ошибке — сообщение о неверных данных.
Регистрация новых пользователей выполняется в классе ClientRegistrationService. Метод register выполняет комплексную валидацию входных данных: проверку длины и формата ФИО и должности, соответствие табельного номера шаблону (одна латинская буква, дефис, четыре цифры), формат телефона (12 цифр в формате 375XXXXXXXXX), политику пароля через класс PasswordPolicy (минимум 6 символов, латинские буквы и цифры). Также проверяется уникальность ФИО, должности, табельного номера, телефона, email и пароля. При обнаружении дубликатов возвращается ответ с перечнем конфликтующих полей. При успешной регистрации формируется идентификатор пользователя с префиксом в зависимости от роли: SU- для сотрудника, ST- для кладовщика, AD- для администратора.
Административное управление пользователями реализовано в SystemUserListController. Эндпоинты модуля позволяют получить список всех пользователей, просмотреть данные конкретного пользователя, создать новую учётную запись, обновить существующую и удалить пользователей по списку идентификаторов. При удалении выполняется проверка наличия связанных заявок, что предотвращает нарушение целостности данных.

Модуль каталога номенклатуры (supply)

Модуль supply обеспечивает работу с каталогом материалов и оборудования. Центральным компонентом является класс SupplyGoodService, который реализует получение, фильтрацию, создание, обновление и удаление товаров. JPA-сущность SupplyGood отображается на таблицу supply_goods, репозиторий SupplyGoodRepository предоставляет методы доступа к данным, включая запросы для получения уникальных категорий, брендов и стран.
Метод getCatalogSummary формирует сводку каталога для главной страницы: список категорий и подкатегорий с количеством товаров в каждой. Метод getMeta возвращает метаданные для фильтров интерфейса — уникальные категории, подкатегории по категориям, бренды и страны производства. Метод findFiltered реализует поиск товаров с учётом параметров фильтрации: категория, подкатегория, текстовый поиск, бренд, страна, минимальная и максимальная цена, а также режим сортировки. Фильтрация реализована через JPA Specifications в классе SupplyGoodSpecifications, что позволяет динамически формировать запросы к базе данных в зависимости от переданных параметров.
Создание и обновление товаров выполняется методами create и update. При создании автоматически генерируется идентификатор формата SG-1001, SG-1002 и т.д. на основе максимального существующего номера. Удаление поддерживает массовое удаление по списку идентификаторов. Контроллер SupplyGoodController предоставляет REST API для всех операций модуля и обрабатывает HTTP-запросы от клиентской части.

Модуль заявок и заказов (order)

Модуль order реализует наиболее сложную бизнес-логику системы — формирование, оформление и обработку заявок на материалы. Центральный класс OrderService управляет полным жизненным циклом заявки. JPA-сущности CustomerOrder и OrderLine отображаются на таблицы orders и order_lines, репозитории CustomerOrderRepository и OrderLineRepository обеспечивают доступ к данным.
Работа с текущей заявкой (корзиной) начинается с метода current, который возвращает черновик заявки пользователя со статусом draft или пустой объект, если черновик ещё не создан. Метод ensureDraftOrder создаёт новый черновик при первом добавлении товара: генерируется идентификатор формата OR-1001, устанавливается статус draft и фиксируются временные метки создания и обновления.
Добавление товара в корзину реализовано методом addItem. Если позиция с данным товаром уже есть в черновике, увеличивается количество; если нет — создаётся новая строка в таблице order_lines с идентификатором OL-1001 и фиксируется цена товара на момент добавления (unit_price). Методы changeQuantity и removeItems позволяют изменять количество и удалять позиции из корзины.
Оформление заявки выполняется методом place. Перед оформлением проверяется, что корзина не пуста; затем статус изменяется с draft на «в ожидании», и заявка становится доступной для обработки кладовщиком. Метод ordersForAccount возвращает список заявок пользователя для личного кабинета, исключая черновики. Метод ordersWorkspace возвращает все оформленные заявки для складской и административной панелей.
Изменение статуса заявки реализовано методом updateStatus, который используется кладовщиком для перевода заявки по цепочке: «принят» → «в обработке» → «собирается» → «готов к получению». Подтверждение получения материалов сотрудником выполняется методом pickup: проверяется, что заявка принадлежит пользователю и находится в статусе «готов к получению», после чего статус изменяется на «получен». Метод counts возвращает количество позиций в корзине и число активных заявок пользователя для отображения счётчиков в интерфейсе.
Контроллер OrderController объединяет все операции модуля в REST API и обрабатывает запросы на получение текущей заявки, добавление и удаление позиций, оформление, просмотр истории, изменение статусов и подтверждение получения.

Модуль конфигурации (config)

Модуль config содержит классы настройки приложения. Класс SecurityConfig конфигурирует Spring Security: отключает CSRF-защиту (необходимо для REST API), настраивает политику CORS для разрешения запросов с клиентского приложения (http://localhost:3000) и разрешает доступ ко всем эндпоинтам без серверной авторизации на уровне Spring Security. Проверка пользователя выполняется на уровне API аутентификации в модуле client.
Класс WebConfig настраивает раздачу статических изображений товаров. Изображения хранятся в каталоге backend/images/ и доступны по URL /images/**, что позволяет клиентской части отображать фотографии товаров из каталога.


Генерация идентификаторов

Во всех модулях используется единый подход к генерации идентификаторов записей. Вместо автоинкрементных числовых ключей применяются строковые идентификаторы с префиксами: SU-, ST-, AD- для пользователей, SG- для товаров, OR- для заявок, OL- для позиций заказа. Новый идентификатор формируется путём поиска максимального существующего номера с данным префиксом и увеличения его на единицу. Такой подход делает идентификаторы читаемыми и соответствующими предметной области.

Обработка ошибок и валидация

Валидация входных данных выполняется на нескольких уровнях. На уровне DTO используются аннотации Jakarta Validation (@Valid) в контроллерах. На уровне сервисов реализована дополнительная бизнес-валидация: проверка форматов, уникальности, допустимости операций (например, запрет оформления пустой заявки). При ошибках сервисы выбрасывают исключения IllegalArgumentException и IllegalStateException, а контроллеры возвращают соответствующие HTTP-коды: 400 Bad Request, 401 Unauthorized, 404 Not Found, 409 Conflict.

Взаимодействие модулей

Модули системы взаимодействуют друг с другом через сервисный слой. Модуль order обращается к репозиториям supply и client для получения данных о товарах и пользователях при формировании заявок. Модуль supply работает автономно при операциях с каталогом. Модуль client используется при аутентификации и администрировании. Такое взаимодействие обеспечивает согласованность данных и корректную работу бизнес-процессов снабжения.
Таким образом, серверная часть на Spring Boot реализует полный набор функций, необходимых для автоматизации процессов снабжения подразделений РУП «Гродноэнерго»: управление пользователями, работу с каталогом номенклатуры, формирование и обработку заявок. Многослойная архитектура, модульная организация кода и использование Spring Data JPA обеспечивают надёжность, расширяемость и удобство сопровождения серверного приложения.

3.4 Реализация клиентской части на React

Клиентская часть веб-приложения реализована как одностраничное приложение (SPA) на библиотеке React 19 с использованием Create React App. Она отвечает за отображение пользовательского интерфейса, обработку действий пользователя, отправку запросов к REST API серверной части и обновление экрана без перезагрузки страницы. Клиентская часть не хранит бизнес-данные постоянно: после каждого значимого действия (вход, добавление в корзину, смена статуса) данные запрашиваются с сервера и отображаются в актуальном виде.

Структура клиентского приложения

Проект frontend организован по функциональному принципу. Точка входа — файл index.js, который монтирует корневой компонент App в DOM-элемент #root с помощью ReactDOM.createRoot. Подключаются глобальные стили index.css и библиотека иконок Font Awesome.

Основные каталоги клиентской части:

•	src/App.js — маршрутизация приложения;
•	src/pages/ — страницы (экраны) системы;
•	src/context/ — глобальное состояние приложения;
•	src/utils/ — вспомогательные функции.

Такое разделение упрощает навигацию по коду: каждая страница отвечает за свой экран, общая логика работы с API и состоянием пользователя сосредоточена в контексте, утилиты используются повторно на разных страницах.

Маршрутизация приложения

Навигация между экранами реализована с помощью React Router 7. В файле App.js определены маршруты:

•	/ — главная страница;
•	/catalog — каталог номенклатуры;
•	/catalog/product/:productId — карточка товара;
•	/sign-in-sign-up — вход и регистрация;
•	/profile/:id — личный кабинет;
•	/current-order/:id — корзина и оформление заявки;
•	/admin-dashboard — административная панель;
•	/warehouse-dashboard — складская панель.

Все маршруты обёрнуты в компонент AppDataProvider, который обеспечивает доступ к глобальному состоянию на любой странице. Используется BrowserRouter, поэтому переходы между разделами выполняются без перезагрузки страницы — меняется только содержимое области контента.
После успешного входа выполняется программное перенаправление в зависимости от роли: сотрудник — в каталог, кладовщик — на складскую панель, администратор — в административную панель. Это реализовано в компоненте AuthPage через хук useNavigate.

Глобальное состояние

Центральным элементом клиентской архитектуры является контекст AppDataContext, реализованный в файле context/AppDataContext.jsx. React Context API позволяет передавать данные и функции через дерево компонентов без проброса props на каждом уровне.

Контекст хранит:

•	currentUser — текущий авторизованный пользователь (id, email, имя, роль);
•	currentOrder — текущая корзина (черновик заявки);
•	cartItems — позиции корзины;
•	orders — список заявок (для личного кабинета, склада или админки);
•	orderCounts — счётчики позиций в корзине и активных заявок;
•	profileInfo — данные профиля пользователя.
Контекст также предоставляет функции для работы с системой:
•	loginDemo / logout — вход и выход;
•	addToCart, changeCartQty, removeCartItems — работа с корзиной;
•	placeOrderFromCart — оформление заявки;
•	updateOrderStatus — смена статуса (для кладовщика);
•	pickUpOrders — подтверждение получения;
•	refreshOrders — обновление списка заявок.

При изменении currentUser автоматически загружаются связанные данные: корзина, заявки, счётчики и профиль. Это реализовано через хук useEffect, который вызывает refreshAllOrders при входе пользователя.
Доступ к контексту на страницах выполняется через хук useAppData(), который возвращает объект состояния и функций. Если хук вызывается вне AppDataProvider, выбрасывается ошибка — это предотвращает использование данных вне контекста приложения.

Взаимодействие с REST API

Обмен данными с серверной частью выполняется через Fetch API. Вспомогательная функция apiFetch внутри AppDataContext формирует HTTP-запросы к backend: устанавливает заголовки Accept и Content-Type, передаёт тело запроса в формате JSON, обрабатывает ответ и выбрасывает ошибку при неуспешном статусе.
Базовый URL API определяется утилитой apiOrigin.js. В режиме разработки запросы направляются на http://localhost:8080, что согласовано с настройкой CORS на сервере. При необходимости адрес можно переопределить через переменную окружения REACT_APP_API_ORIGIN.
Данные, полученные от сервера, проходят нормализацию перед сохранением в состоянии. Функция normalizeOrder приводит структуру заявки к единому формату: преобразует идентификаторы, числовые поля, пути к изображениям через backendAssetUrl, вычисляет общее количество и сумму. Функция normalizeProfile формирует единообразное представление профиля пользователя.

Реализация страниц

Каждая страница приложения — самостоятельный React-компонент в каталоге pages/.
HomePage — информационный лендинг с описанием платформы «Гродноэнерго» и сводкой каталога. При загрузке выполняется запрос к /api/supply-goods/catalog-summary, результат отображается в виде таблицы категорий и подкатегорий с количеством товаров.
AuthPage объединяет формы входа и регистрации. Компоненты SignIn и SignUp отправляют данные на /api/auth/login и /api/auth/register. На клиенте выполняется предварительная валидация через утилиты authValidators.js и tabNumberInput.js (формат табельного номера, телефона, пароля).
CatalogPage — основной экран каталога. Использует хуки useState, useEffect, useMemo и useCallback для управления фильтрами, загрузки метаданных и списка товаров. Реализованы текстовый поиск с подсветкой совпадений, фильтры по категории, подкатегории, бренду, стране и цене, а также сортировка.
ProductPage загружает данные одного товара по идентификатору из URL-параметра productId и отображает карточку с возможностью добавления в корзину через addToCart из контекста.
CurrentOrderPage отображает корзину: список позиций, изменение количества, удаление, общую сумму и кнопку оформления заявки. Взаимодействует с контекст через cartItems, changeCartQty, removeCartItems, placeOrderFromCart.
AccountPage показывает профиль и заявки пользователя: активные и завершённые, с визуализацией статусов и возможностью подтверждения получения через pickUpOrders.
WarehouseDashboard — рабочее место кладовщика: список заявок из /api/orders/workspace, поиск, смена статусов через updateOrderStatus, автоматическое обновление данных по интервалу.
AdminDashboard — панель администратора с вкладками управления пользователями и товарами. Загружает данные из /api/system-users и /api/supply-goods, поддерживает CRUD-операции через соответствующие REST-эндпоинты.

Вспомогательные утилиты

Каталог utils/ содержит переиспользуемые функции:

•	apiOrigin.js — формирование абсолютного URL для API;
•	backendAssetUrl.js — преобразование путей к изображениям товаров;
•	authValidators.js — валидация полей форм входа и регистрации;
•	tabNumberInput.js — маска и нормализация табельного номера;
•	passwordDiagnostics.js — диагностика пароля в административной панели.

Выделение утилит в отдельные модули снижает дублирование кода и упрощает сопровождение.

Компонентный подход и интерфейс

Клиентская часть построена на компонентном подходе React. Страницы могут включать локальные компоненты (например, LogoutFloatingButton — кнопка выхода, закреплённая в правом верхнем углу). Состояние интерфейса (фильтры, выбранные элементы, режимы форм) управляется через useState, производные значения — через useMemo, обработчики — через useCallback для оптимизации перерисовок.
Стилизация выполнена с помощью CSS-файлов, привязанных к страницам (catalog.css, HomePage.css, auth.css и др.), и inline-стилей для отдельных элементов. Иконки Font Awesome используются для наглядности кнопок и информационных блоков.

Обработка ролей на клиенте

На клиенте роли пользователей представлены значениями client, warehouse и admin. При входе роль storekeeper с backend преобразуется в warehouse для интерфейса. В зависимости от роли меняется набор доступных страниц и источник данных для списка заявок: сотрудник видит только свои заявки (/api/orders?userId=), кладовщик и администратор — все оформленные заявки (/api/orders/workspace).

Особенности реализации

При реализации клиентской части учтены следующие моменты. Состояние пользователя хранится в памяти React-приложения (без JWT и cookies) — при перезагрузке страницы требуется повторный вход. Все операции с заявками выполняются через REST API, клиент не изменяет данные локально без синхронизации с сервером. Ошибки сетевых запросов обрабатываются через try/catch и отображаются пользователю в виде сообщений.
Таким образом, клиентская часть на React реализует полноценный пользовательский интерфейс системы снабжения: маршрутизацию, глобальное состояние, взаимодействие с REST API и экраны для всех ролей. Компонентная архитектура, централизованный контекст и модульная организация кода обеспечивают удобство разработки и согласованность с серверной частью и функциональными модулями, описанными в следующей подглаве.

3.5 Реализация функциональных модулей приложения

Реализация функциональных модулей является завершающим этапом практической части дипломной работы. Если серверная часть на Spring Boot обеспечивает бизнес-логику и доступ к данным, а клиентская часть на React — общую структуру интерфейса и маршрутизацию, то функциональные модули объединяют обе части в единую систему, понятную конечному пользователю. Каждый модуль соответствует определённому разделу приложения и реализует отдельный фрагмент процесса снабжения: от первого знакомства с платформой до оформления заявки, её обработки на складе и администрирования справочной информации.
Все модули построены по единому принципу: React-страница отображает данные, AppDataContext хранит состояние и вызывает REST API, серверная часть выполняет операции и сохраняет результат в PostgreSQL. Такое разделение позволяет независимо развивать интерфейс и backend, сохраняя согласованность данных между экранами.

Модуль главной страницы

На основе проектных решений, разработанных во второй главе, создано работоспособное программное обеспечение, включающее серверную часть на Spring Boot, клиентскую часть на React и базу данных PostgreSQL. Реализация охватила все основные этапы: от подготовки среды разработки до создания функциональных модулей системы.
На этапе инициализации проекта была подготовлена программная среда и создана структура приложения. Установлены и настроены Java 21, Apache Maven, Node.js, npm и PostgreSQL. Сформированы две части проекта: backend на Spring Boot 3.4.5 и frontend на React 19. Настроено взаимодействие между компонентами: CORS на сервере, proxy и прямое обращение к API на клиенте, подключение к базе данных через application.properties. Создана и настроена база данных PostgreSQL. При запуске backend Hibernate автоматически сформировал четыре таблицы: system_users supply_goods, orders и order_lines. Между таблицами установлены связи один-ко-многим, обеспечивающие целостность данных. Таблицы наполнены тестовыми данными: пользователями с разными ролями, номенклатурой электротехнических товаров, заявками и позициями заказов. Пароли пользователей хранятся в зашифрованном виде с использованием алгоритма bcrypt, что повышает безопасность хранения учётных данных.
Реализована серверная часть на Spring Boot с многослойной архитектурой. Код организован в модули client, supply, order и config. Каждый модуль включает controller, service, repository, entity и dto. REST API обеспечивает операции аутентификации и управления пользователями, работы с каталогом номенклатуры и полного цикла обработки заявок — от формирования черновика до подтверждения получения материалов. Бизнес-логика реализована в сервисных классах с использованием транзакций Spring, валидации входных данных и генерации строковых идентификаторов с префиксами SU-, SG-, OR-, OL-.
Разработана клиентская часть как одностраничное приложение на React. Реализована маршрутизация через React Router, централизованное управление состоянием через AppDataContext, взаимодействие с REST API через Fetch. Созданы страницы для всех ролей пользователей: главная, каталог, карточка товара, вход и регистрация, личный кабинет, корзина, административная и складская панели. Реализованы функциональные модули: аутентификация и регистрация, каталог с фильтрацией и поиском, формирование и оформление заявок, личный кабинет сотрудника, складская и административная панели.
