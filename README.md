# Supply Management System for RUP Grodnoenergo

**Веб-приложение для цифровизации процессов снабжения подразделений РУП «Гродноэнерго»**

Дипломный проект: централизованный каталог номенклатуры, оформление и отслеживание заявок на материалы, складская обработка и администрирование справочников в единой клиент–серверной системе.

## О проекте

Система предназначена для автоматизации внутреннего материально-технического снабжения энергетического предприятия. Она заменяет разрозненные способы оформления заявок (бумажные бланки, электронные таблицы, телефон, электронная почта) единой цифровой платформой.

**Цель:** обеспечить сотрудникам подразделений удобный доступ к каталогу материалов и оборудования, кладовщикам — инструмент обработки заявок, администраторам — управление пользователями и номенклатурой.

**Объект автоматизации:** отдел снабжения филиалов РУП «Гродноэнерго».

---

## Ключевые возможности

### Для сотрудника подразделения
- просмотр главной страницы со сводкой каталога по категориям и подкатегориям;
- вход и регистрация с проверкой учётных данных;
- каталог номенклатуры с поиском, фильтрами (категория, подкатегория, бренд, страна, цена) и сортировкой;
- карточка товара с добавлением в корзину;
- формирование черновика заявки и оформление заказа;
- личный кабинет с активными и завершёнными заявками;
- подтверждение получения материалов при статусе «готов к получению».

### Для кладовщика
- складская панель с просмотром оформленных заявок;
- поиск по ФИО, должности и табельному номеру заказчика;
- индикатор прогресса обработки;
- последовательная смена статусов заявки;
- автоматическое обновление списка заявок.

### Для администратора
- управление пользователями (CRUD);
- управление номенклатурой товаров (CRUD);
- поиск и сортировка в таблицах;
- сводные показатели (количество пользователей, заявок, товаров);
- маскированное отображение паролей в списке.

## Архитектура

Приложение построено по модели **клиент–сервер**:

```
Пользователь
    ↓
React SPA (браузер, порт 3000)
    ↓ HTTP / JSON
REST API — Controller (Spring Boot, порт 8080)
    ↓
Service (бизнес-логика)
    ↓
Repository (Spring Data JPA)
    ↓ SQL
PostgreSQL
    ↓
JSON-ответ → React обновляет интерфейс
```

### Серверные модули

| Модуль | Назначение |
|--------|------------|
| `client` | регистрация, аутентификация, управление пользователями |
| `supply` | каталог номенклатуры, фильтрация, метаданные |
| `order` | корзина, оформление заявок, статусы, подтверждение получения |
| `config` | CORS, Spring Security, раздача статических изображений |

Каждый модуль организован по слоям: **controller → service → repository → entity → dto**.

### Клиентская часть

- одностраничное приложение (SPA) на React;
- централизованное состояние через `AppDataContext`;
- маршрутизация через React Router без перезагрузки страницы;
- перенаправление пользователя в нужный раздел после входа в зависимости от роли.

- `AD-` — администратор (Admin)
- `SG-` — товар (Supply Good)
- `OR-` — заявка (Order)
- `OL-` — позиция заявки (Order Line)

## Жизненный цикл заявки

```
draft (черновик)
    ↓ оформление сотрудником
«в ожидании»
    ↓ кладовщик
«принят» → «в обработке» → «собирается» → «готов к получению»
    ↓ подтверждение сотрудником
«получен»
```

| Статус | Описание |
|--------|----------|
| `draft` | черновик в корзине, не виден кладовщику |
| `в ожидании` | заявка оформлена и ожидает принятия |
| `принят` | заявка принята кладовщиком |
| `в обработке` | выполняется обработка |
| `собирается` | идёт комплектация материалов |
| `готов к получению` | материалы готовы к выдаче |
| `получен` | сотрудник подтвердил получение |

## База данных

PostgreSQL, 4 таблицы со связями **один-ко-многим**:

| Таблица | Назначение |
|---------|------------|
| `system_users` | учётные записи: ФИО, должность, табельный номер, телефон, email, пароль, роль |
| `supply_goods` | номенклатура: наименование, категория, подкатегория, бренд, страна, цена, единица измерения, изображение |
| `orders` | заявки: пользователь, статус, даты создания и обновления |
| `order_lines` | позиции заявки: товар, количество, `unit_price` (цена на момент добавления) |

**Связи:**
- один пользователь → много заявок;
- одна заявка → много позиций;
- один товар → может входить в разные заявки.

Hibernate автоматически создаёт и обновляет схему при запуске (`spring.jpa.hibernate.ddl-auto=update`).

## REST API

Базовый URL backend: `http://localhost:8080`

### Аутентификация и пользователи

| Метод | Путь | Назначение |
|-------|------|------------|
| POST | `/api/auth/login` | вход в систему |
| POST | `/api/auth/register` | регистрация |
| GET | `/api/system-users` | список пользователей |
| GET | `/api/system-users/{id}` | профиль пользователя |
| POST | `/api/system-users` | создание пользователя |
| PUT | `/api/system-users/{id}` | обновление пользователя |
| DELETE | `/api/system-users` | удаление пользователей |

### Каталог номенклатуры

| Метод | Путь | Назначение |
|-------|------|------------|
| GET | `/api/supply-goods` | список товаров с фильтрами |
| GET | `/api/supply-goods/meta` | метаданные для фильтров и форм |
| GET | `/api/supply-goods/catalog-summary` | сводка каталога для главной страницы |
| GET | `/api/supply-goods/item/{id}` | карточка товара |
| POST | `/api/supply-goods` | добавление товара |
| PUT | `/api/supply-goods/{id}` | изменение товара |
| DELETE | `/api/supply-goods` | удаление товаров |

### Заявки

| Метод | Путь | Назначение |
|-------|------|------------|
| GET | `/api/orders/current` | текущий черновик (корзина) |
| POST | `/api/orders/current/items` | добавить товар в корзину |
| PATCH | `/api/orders/current/items/{supplyGoodId}` | изменить количество |
| DELETE | `/api/orders/current/items` | удалить позиции |
| POST | `/api/orders/current/place` | оформить заявку |
| GET | `/api/orders?userId=` | заявки пользователя |
| GET | `/api/orders/workspace` | заявки для складской панели |
| GET | `/api/orders/counts` | счётчики корзины и заявок |
| PATCH | `/api/orders/{orderId}/status` | смена статуса (кладовщик) |
| POST | `/api/orders/pickup` | подтверждение получения (сотрудник) |

## Установка и запуск

### 1. Клонирование репозитория

```bash
git clone https://github.com/Anton6000/supply-grodnoenergo.git
cd supply-grodnoenergo
```

### 2. База данных PostgreSQL

1. Установите и запустите PostgreSQL.
2. Создайте базу данных (по умолчанию используется `postgres`):

```sql
-- при необходимости создайте отдельную БД:
CREATE DATABASE grodnoenergo_supply;
```

3. Проверьте параметры подключения в `backend/src/main/resources/application.properties`:

```properties
spring.datasource.url=jdbc:postgresql://localhost:5432/postgres
spring.datasource.username=postgres
spring.datasource.password=root
```

При первом запуске backend Hibernate автоматически создаст таблицы. Данные можно добавить через регистрацию в приложении или административную панель.

### 3. Запуск backend

```bash
cd backend
mvn clean install
mvn spring-boot:run
```

Backend будет доступен на **http://localhost:8080**

### 4. Запуск frontend

В **новом** терминале:

```bash
cd frontend
npm install
npm start
```

Frontend откроется на **http://localhost:3000**

### 5. Проверка работы

1. Откройте http://localhost:3000 — главная страница.
2. Перейдите на `/sign-in-sign-up` — зарегистрируйтесь или войдите.
3. Сотрудник: каталог → товар → корзина → оформление → личный кабинет.
4. Кладовщик: складская панель → смена статусов.
5. Администратор: панель управления → пользователи и товары.

## Конфигурация

### Backend (`application.properties`)

| Параметр | По умолчанию | Описание |
|----------|--------------|----------|
| `server.port` | `8080` | порт REST API |
| `spring.datasource.url` | `jdbc:postgresql://localhost:5432/postgres` | URL БД |
| `spring.datasource.username` | `postgres` | пользователь БД |
| `spring.datasource.password` | `root` | пароль БД |
| `app.cors.allowed-origins` | `http://localhost:3000` | разрешённые origins для CORS |

### Frontend

Создайте файл `frontend/.env` (не коммитится в git):

```env
REACT_APP_API_ORIGIN=http://localhost:8080
```

Если переменная не задана, в режиме development клиент обращается к `http://localhost:8080` напрямую. В `package.json` также настроен proxy на backend.

### Форматы данных при регистрации

| Поле | Формат |
|------|--------|
| Табельный номер | одна латинская буква, дефис, 4 цифры (например, `T-1111`) |
| Телефон | 12 цифр, формат `375XXXXXXXXX` |
| Пароль | минимум 6 символов, латинские буквы и цифры |

## Безопасность

- пароли хранятся в базе данных в зашифрованном виде (bcrypt через pgcrypto PostgreSQL);
- Spring Security настроен для CORS и политики доступа;
- проверка учётных данных выполняется на уровне API аутентификации;
- валидация входных данных на клиенте (React) и сервере (Spring Validation);
- разграничение функциональности по ролям в интерфейсе.

> **Примечание:** в учебной версии проекта состояние пользователя хранится в памяти React-приложения (без JWT). При перезагрузке страницы требуется повторный вход. Полноценная серверная авторизация на уровне API — направление дальнейшего развития.
